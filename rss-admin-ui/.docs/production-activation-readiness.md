# Production Activation Readiness Contract

Status: `MS-020D_PRODUCTION_READINESS_PACKAGED_NO_DEPLOY - NOT_DEPLOYED`.

MS-020D packages the `rss-admin-ui` production activation readiness contract and local evidence harness. It does not deploy, restart, pull, publish an image, activate DNS/TLS/OpenLiteSpeed routing, or contact production. Production mutation remains `PRODUCTION_MUTATION_NOT_PERFORMED`; the admin UI remains `ADMIN_UI_NOT_DEPLOYED`; Tenant/admin login and browser session implementation remain `AUTH_SESSION_DEFERRED`.

## Purpose And Persona

The current UI persona is an unauthenticated read-only observer of coarse service health. It is suitable only for a status-only dashboard whose fields are already available through the backend public health routes and whose future operator explicitly accepts public visibility before activation.

This contract does not authorize business pages, writes, Agent operations, Tenant administration, privileged troubleshooting output, login, logout, refresh tokens, session cookies, or user-specific data. Those require a separately authorized milestone.

## Readiness Decision

The current status-only dashboard may be activated as a public status surface only if the operator authority record accepts the public-data classification below and the deployed runtime matches the exact transport contract. Tenant/admin auth is not required for these current fields because they are status-only and already exposed by backend health endpoints. Auth becomes mandatory before any new field, route, business object, operator action, tenant detail, user detail, or credential-bearing API is exposed.

Any backend response shape change, frontend field expansion, or unsafe display value must fail closed, be redacted, or be classified as `AUTH_OR_REDACTION_REQUIRED_BEFORE_PRODUCTION` before activation.

## Data Classification

| Field | Source route | Already public through backend health | Secret, personal, or business data | Display without auth | Fallback or redaction if shape changes |
|---|---|---:|---:|---:|---|

| Redis dependency state | `dependencies.redis` from `/health/ready` | Yes, as coarse dependency status only | No, while limited to up/down/unknown style values | Yes | Any endpoint, key, queue, metric, credential, tenant, or diagnostic detail becomes `AUTH_OR_REDACTION_REQUIRED_BEFORE_PRODUCTION` |

## Future Production Transport Contract

Planned browser origin: `https://rss-panel.habersoft.com` (`OPERATOR_AUTHORIZED_FUTURE_TASK_ONLY`, planned and inactive in MS-020D).

Browser routes are exact and relative:

```text
GET /status-api/health/live
GET /status-api/health/ready
```

The browser must not receive, configure, or infer the upstream backend origin. `ADMIN_UI_HEALTH_UPSTREAM_ORIGIN` is server-only runtime config. `ADMIN_UI_API_BASE_URL` is disallowed for this health slice and must not appear in browser config, static assets, HTML, or UI output. Hardcoded production backend hosts are disallowed in browser code.

The frontend runtime may map only:

```text
/status-api/health/live  ->  <ADMIN_UI_HEALTH_UPSTREAM_ORIGIN>/health/live
/status-api/health/ready ->  <ADMIN_UI_HEALTH_UPSTREAM_ORIGIN>/health/ready
```

Disallowed transport behaviors:

- generic `/api/*` or query-controlled proxying,
- forwarding browser query strings to health upstreams,
- forwarding browser request bodies,
- forwarding browser credentials, cookies, bearer tokens, Tenant tokens, Agent keys, or arbitrary client headers,
- relaying upstream `Set-Cookie` or `WWW-Authenticate`,
- browser-visible `ADMIN_UI_HEALTH_UPSTREAM_ORIGIN` or `ADMIN_UI_API_BASE_URL`,
- backend CORS changes for this health slice,
- backend source, route, Prisma, package, version, or production env changes.

## Future Edge And Server Activation Requirements

All items in this section are `OPERATOR_AUTHORIZED_FUTURE_TASK_ONLY`; none are active in MS-020D.

- `rss-panel.habersoft.com` remains planned, not active.
- The edge should terminate TLS and route only to the admin UI loopback frontend port.
- The admin UI container should bind only to loopback on the host-facing side.
- The frontend container should reach the backend only through server-only `ADMIN_UI_HEALTH_UPSTREAM_ORIGIN`, expected to be a loopback or private server-side origin selected by the operator.
- The backend must not be changed to add CORS for the health dashboard.
- TLS certificates, HTTP-to-HTTPS redirect, HSTS, access logging, error logging, firewall rules, and operational monitoring are future operator tasks.
- Future activation must use an immutable image identity or an explicitly recorded local build identity policy accepted by the operator.

## Local Readiness Evidence Harness

Run from `rss-admin-ui`:

```bash
npm run verify:production-readiness
```

The verifier is local-only. It builds the production frontend bundle, builds `rss-admin-ui:ms020d-local` by default, proves missing and invalid upstream origins fail closed, reuses the proxy-security harness with a synthetic Docker-network sentinel, validates root and frontend production Compose templates with synthetic values, and refuses known production hostnames in its generated command environments.
